# COMS10017 OOP & Java Basics README

## 1. 这部分课程到底在讲什么

这几份讲义的核心目标，是带你从 **C 风格的过程式编程** 过渡到 **面向对象编程（Object-Oriented Programming, OOP）**，再进一步认识为什么课程选择 **Java** 作为主要语言。课程强调：软件不只是“能跑的一段程序”，而是对现实世界或虚拟世界中事物、属性、行为的建模；而当系统变大、多人协作、平台复杂、还要求可靠和易维护时，就需要更合适的语言与工具支持。

---

## 2. 从 C 到 OOP：为什么要换一种思路

在 C 里，你之前主要做的是过程式编程：

* 定义类型和变量
* 写函数
* 用参数把数据传给函数
* 用指针控制内存
* 把代码拆到不同文件里

这种方式适合小程序，也能完成很多计算任务，但它有一个明显特点：**数据本身是被动的，函数可以作用在任何数据上**。当程序越来越大时，这种“数据和操作分离”的方式会让组织、复用、维护变得更难。课件里因此提出了 OOP 的几个目标：

* **Modularity（模块化）**：代码拆成清晰的小单元
* **Reusability（可复用）**：避免重复写相同逻辑
* **Maintainability（可维护）**：方便加功能、改旧代码
* **Low Coupling, High Cohesion（低耦合，高内聚）**：模块之间尽量独立，模块内部逻辑尽量集中
* **Scalability（可扩展）**：小组件能组合成更复杂的大系统

同时，课件把这些目标和 OOP 中的重要思想对应起来：

* **Abstraction** → classes / interfaces
* **Inheritance** → subclassing
* **Polymorphism** → overriding
* **Encapsulation** → access modifiers
* **Generics** → type parameters

也就是说，OOP 不是为了“写得花”，而是为了管理复杂度。

## 3. 什么是对象（Object）

对象是 OOP 的中心概念。课件里对对象的定义可以概括成三点：

### 3.1 Behaviour（行为）

对象能“做什么”。

比如一个机器人对象可以：

* `talk("Hello")`
* `charge(1.0)`

也就是对象带着自己的方法，可以执行动作。

### 3.2 Attributes（属性）

对象“是什么”。

比如一个机器人对象可以有：

* `name`
* `numLegs`
* `powerLevel`

这些属性描述了对象当前的状态。

### 3.3 Identity（身份）

对象是**独一无二的实体**。

即使两个对象属性一模一样，它们仍然不是同一个对象。课件举的理解方式是：对象有自己的唯一身份，比如运行时内存中的一个位置。

### 3.4 一句话总结对象

**对象 = 数据（attributes / state）+ 方法（behaviour）+ 唯一身份（identity）**

对象不是单纯一堆数据，也不是单纯一组函数，而是把“数据”和“操作这些数据的方法”打包在一起。

---

## 4. 什么是类（Class）

如果对象是“具体东西”，那么类就是“蓝图（blueprint）”。

课件把类解释为：**每个对象都属于某个类，都是某个类的实例（instance）**。比如：

* `Robot` 是类
* `c3po` 是 `Robot` 类创建出来的对象

类中通常定义：

* **字段（fields）**：保存对象属性
* **方法（methods）**：定义对象行为

例如：

* `String name`
* `int numLegs`
* `float powerLevel`
* `void talk(String)`
* `void charge(float)`

类不只是一个“模板”，它同时还扮演很多角色：

* 它是变量和对象的类型
* 它是一个结构，里面有字段
* 它是一个上下文，让方法天然知道自己属于谁
* 它是一个模块，可以单独编译
* 它是一个命名空间，避免名字冲突
* 它也是设计与测试的基本单位

所以类在 Java 里既是“蓝图”，也是“组织代码的单位”。

---

## 5. Class 和 Object 的区别

这是最容易混的点之一。

### Class

* 是类别
* 是编译期概念
* 是蓝图
* 更偏静态
* 描述“这一类对象应该长什么样、会做什么”

### Object

* 是实例
* 是运行期概念
* 是实体
* 更偏动态
* 真正带着当前状态活在程序里

可以这样记：

> **class 像建筑图纸，object 像按图纸建出来的一栋具体房子。**

图纸本身不住人，房子才住人；类本身不“说话”，对象才会在运行时真正调用方法。

---

## 6. 类是怎么写出来的

课件用 `Robot` 作为例子，展示了一个类最基本的结构：

* 字段：保存状态
* 方法：操作状态

比如 `talk()` 会检查 `powerLevel` 是否足够；如果够，就输出说的话，并消耗电量；如果不够，就提示太弱不能说话。`charge()` 会增加电量。这个例子说明了一件很重要的事：

> 方法不是随便处理任意数据，而是**处理它自己对象内部的数据**。

这和 C 里“函数 + 外部数据”的感觉不一样。在 Java OOP 里，行为和状态被装进同一个对象里。

## 7. 构造器（Constructor）与实例化（Instantiation）

### 7.1 什么是实例化

实例化就是：**根据类创建对象**。

Java 用 `new` 来做这件事：

```java
Robot c3po = new Robot("C3PO");
```

这里发生的事是：

1. `Robot` 类作为蓝图已经存在
2. `new Robot("C3PO")` 创建一个新对象
3. 构造器初始化它的内部状态
4. 变量 `c3po` 用来引用这个对象

### 7.2 什么是构造器

构造器是“对象初始化的方法”。

特点：

* 名字和类名相同
* 没有返回类型
* 在 `new` 的时候自动调用

例子里：

```java
Robot(String productName) {
    name = productName;
    numLegs = 2;
    powerLevel = 2.0f;
}
```

这说明构造器的作用就是：**让对象一出生就处于一个合理的初始状态**。

### 7.3 为什么构造器很重要

因为对象不能“半死不活”地存在。一个新建出来的机器人，至少应该知道自己叫什么、几条腿、初始电量多少。

所以可以这样理解：

> **构造器 = 对象的出生流程。**

---

## 8. main 方法：程序入口

Java 程序通常从下面这个方法开始执行：

```java
public static void main(String[] args){}
```

它是整个程序的入口点。课件里的 `RobotWorld` 类就是用 `main` 创建 `Robot` 对象并调用方法：

```java
Robot c3po = new Robot("C3PO");
c3po.talk("Hello, Java!");
```

这说明：

* `main` 负责启动程序
* 在 `main` 里你通常会创建对象、组织对象之间的交互
* 真正的业务行为应该尽量放在对象的方法里，而不是都塞进 `main` 里

---

## 9. 点号（dot notation）到底表示什么

Java 里经常写：

```java
c3po.talk("Hello");
c3po.name
```

点号 `.` 的含义是：

> **访问某个对象的成员（字段或方法）**

所以：

* `c3po.talk(...)` 是调用 `c3po` 这个对象的方法
* `c3po.name` 是访问 `c3po` 这个对象的字段

这也体现了“对象有自己的数据和行为”。

---

## 10. 可见性修饰符（Visibility Modifiers）与封装（Encapsulation）

课件很快提到一个重要概念：**封装**。

示例里把字段改成了：

```java
private String name;
private int numLegs;
private float powerLevel;
```

方法则写成：

```java
public void talk(String phrase)
public void charge(float amount)
```

这表示：

* `private`：只能在类内部访问
* `public`：可以从类外部访问

这样做的意义是：

1. **保护对象内部状态**
2. 不让外部随便乱改字段
3. 让对象自己负责管理自己的数据
4. 暴露给外部的是“能做什么”，而不是“内部怎么存”

这就是封装的核心：**隐藏实现细节，只暴露必要接口**。

---

## 11. 什么是 OOP

OOP 是一种基于“对象”概念的编程范式。对象属于类，包含数据（attributes）和过程代码（methods），程序由一组互相调用方法、彼此交互的对象组成。

用更直白的话说：

> **OOP 不是先想“我要写哪些函数”，而是先想“系统里有哪些对象，它们各自负责什么，它们怎么合作”。**

---

## 12. 软件结构的三种思路

课件还对比了三种组织软件的方式：

### 12.1 Unstructured

比如很早期的汇编风格，代码和流程混在一起，难以维护。

### 12.2 Procedural / Modular

比如 C / Pascal，程序由函数和模块组成，已经比无结构程序好很多。

### 12.3 Object-Oriented

程序由带状态、带身份的对象组成，对象之间协作完成任务。

这个对比很重要，因为它说明：

* OOP 不是凭空冒出来的
* 它是为了应对更大规模、更复杂的软件设计问题
* 但它也不是“唯一正确答案”

---

## 13. OOP 也不是完美的：课件里的批判

课件没有把 OOP 讲成“神话”，而是很诚实地提了批评点：

* 让“名词/对象”主导程序结构，可能会过度强调数据而忽视算法
* 类和固定关系有时会带来僵硬性
* OO 程序常常形成庞大的状态对象网络，复杂度会上升
* 某些场景下，声明式代码可能更简洁
* OO 正确性的形式化证明工具还不够成熟

这部分很值得记住，因为它提醒你：

> **学习 OOP 不是为了迷信 OOP，而是理解它适合解决什么问题，不适合什么问题。**

## 14. 为什么课程要学 Java

课件给了很多理由：

* Java 是使用非常广泛的语言
* 它天生支持面向对象
* 它平台独立
* 有丰富的 API 和库
* 有垃圾回收（Garbage Collection）
* 有较好的文档支持（JavaDocs）
* 有强大的开发工具，比如 IntelliJ
* 免费且社区大

同时还强调，Java 在大型系统、设计、文档、可移植性、可扩展性方面表现很强；而 C 更强在效率和内存控制，Haskell 更强在算法与正确性。也就是说，Java 是课程里一个比较平衡、适合教学 OOP 和工程化开发的选择。

## 15. Java 和 C 的关系

Java 不是和 C 完全没关系。课件指出，两者共享很多语法：

* 基本类型
* 运算符
* `if / else / switch / while / for`
* 函数调用形式
* 花括号、分号
* 注释风格

所以从 C 转 Java 时，你会觉得很多语法长得很像。

但 Java 和 C 也有关键区别：

* Java 强调 **classes and objects**
* Java 有 **exception handling**
* Java 有 **garbage collection**
* Java **platform independent**
* Java 有默认值机制
* Java 没有像 C 那样暴露给你的裸指针编程方式

所以表面语法相似，但编程思维差别很大。课件甚至专门提醒：不要把 Java 用成“披着 Java 皮的 C”。

## 16. Java 工具链（Toolchain）

这是写 Java 必须理解的基础流程。

### 16.1 JDK

Java Development Kit。开发工具包，里面包含编译器等工具。`javac` 属于这里。

### 16.2 JVM

Java Virtual Machine。负责运行 `.class` 字节码。`java` 命令会通过 JVM 执行程序。

### 16.3 JRE

Java Runtime Environment。提供程序运行时需要的标准库支持。

### 16.4 编译与运行流程

Java 的基本流程是：

1. 写 `.java` 源文件
2. 用 `javac` 编译成 `.class`
3. 用 `java` 通过 JVM 运行

比如：

```bash
javac RobotWorld.java
java RobotWorld
```

课件还区分了两类问题：

* **编译期错误（compiler errors）**：`javac` 阶段报错
* **运行期异常（runtime exceptions）**：`java` 阶段才发生

这点非常重要，因为以后调试时你要先分清：是“根本没编译过”，还是“编过了但运行崩了”。

## 17. static：先认识，但不要滥用

课件对 `static` 的态度非常明确：**先认识它，但在 OOP 里不要乱用。**

### 17.1 static 是什么

`static` 成员属于**类本身**，而不是某个具体对象。

比如 `main` 是 `static`，因为程序启动时还没有对象，需要先靠类本身提供入口。

### 17.2 static 的特点

* 可以是字段、方法、代码块、嵌套类
* 所有实例共享同一份 static 成员
* 不需要创建对象也能访问
* static 方法不能直接访问非 static 成员

### 17.3 为什么课件说“resist them”

因为如果你把所有东西都写成 `static`，你就退回到了“C 风格的函数 + 全局数据”的思路，等于错过了 Java OOP 的重点。

可以这样理解：

> `static` 不是不能用，而是它代表“属于类”，不是“属于对象”。
> 如果你满屏 static，说明你没有真正让对象承担责任。

## 18. 命令行参数、解析与一个简单 Adder 例子

课件用 `Adder` 和 `Calculator` 演示了一个很基础但很实用的模式：

### `Adder`

* 有一个字段 `sum`
* 构造器把它初始化为 0
* `add(int)` 方法负责累加

### `Calculator`

* 在 `main` 里创建 `Adder`
* 遍历 `args`
* 用 `Integer.parseInt(arg)` 把字符串转成整数
* 再调用 `adder.add(...)`

最后输出总和。

这个例子让你学到几件事：

1. 命令行参数 `args` 本质上是字符串数组
2. 需要解析后才能变成数字
3. 一个对象可以负责保存状态（这里是 `sum`）
4. `main` 只是协调者，真正的数据逻辑放进对象里更合理

## 19. 异常（Exceptions）

Java 的一个重要特性，是它有明确的异常处理机制。

### 19.1 什么是异常

当程序中发生“非正常情况”时，比如解析失败，这就叫异常。它不一定意味着程序必须立刻崩掉。

### 19.2 try-catch

基本结构：

```java
try {
    // 可能出错的代码
} catch (Exception e) {
    // 出错后怎么处理
}
```

意思就是：

* `try` 里放可能出问题的代码
* 如果真的出错，就跳到 `catch`
* `e` 这个对象里带着异常信息

### 19.3 异常处理示例

课件用 `ExceptionalCalculator` 演示：当输入里出现 `two` 而不是数字时，程序不会直接崩溃，而是打印一条提示信息。

### 19.4 throw 和 throws

* `throw`：你自己主动抛出一个异常
* `throws`：方法不自己处理，而是把异常交给调用者处理

这体现了 Java 对错误处理的一种工程化思路：**错误不是被忽略，而是被显式表达和传递。**

## 20. Exit Codes（退出码）

程序结束时会返回一个退出码给操作系统：

* `0`：通常表示成功
* 非零：通常表示发生了错误

例如：

```java
System.exit(0);
System.exit(137);
```

这在命令行、脚本、自动化工具里很重要，因为外部系统往往通过退出码判断程序有没有成功运行。

## 21. 这几份讲义里最该建立的思维模型

### 模型 1：对象不是“数据包”，而是“会管理自己状态的实体”

对象里面既有属性，也有方法。方法不是漂在外面的，而是和对象绑在一起。

### 模型 2：类不是对象，类是对象的蓝图

`Robot` 不是机器人本身，`c3po` 才是具体机器人。

### 模型 3：构造器是“出生初始化”

对象一被 `new` 出来，就要通过构造器进入合法初始状态。

### 模型 4：封装是“把该藏的藏起来”

对象应该自己负责自己的数据，不要让外部随便碰内部字段。

### 模型 5：main 是入口，不是所有逻辑的垃圾桶

`main` 应该负责启动和组织，复杂逻辑应该进类和对象。

### 模型 6：Java 不是 C 的替身

虽然语法像，但思维重心不同：Java 强调对象、异常、平台独立和工程化开发。
